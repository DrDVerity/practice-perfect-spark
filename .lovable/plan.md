# Dedupe Campaign Research Reports & Prevent Duplicates

## Goal

Stop accumulating near-duplicate "Campaign Research: …" reports in each client's Knowledge Base. Keep only the most recent report per (campaign, focus, target audience), and only generate a new one if the campaign focus or target market has actually changed.

## Scope

Two report families are affected:

1. **Campaign Research: {Campaign Name} (date)** — generated by the campaign agent on every full strategy request.
2. **Practice Intelligence Report - {Practice}** — already has 30-day caching (kept as-is, no change needed).

Only #1 is the source of the duplicates the user is seeing (e.g. 5 "Campaign Research: SNS" reports in DDF KB).

## Changes

### 1. Backfill cleanup (one-time)

Run a SQL pass on `knowledge_base` that, for each (`user_id`, normalized campaign name from title) group of `metadata.source = 'campaign-agent-research'` docs:

- Keep the **most recent** row.
- Delete the rest (regardless of age — they're confirmed duplicates because they share campaign + owner and the agent doesn't write focus/audience into the title).

User will see exactly one "Campaign Research: SNS" row in DDF KB after this runs.

### 2. Tag future reports with focus + audience

Update `supabase/functions/campaign-agent/index.ts` → `saveResearchToKB`:

- Accept `campaignFocus` and `targetAudience` from the request body (already pulled from the campaign / profile by the dialog).
- Store them in `metadata` alongside the existing fields.
- Use a **stable title** without the date: `Campaign Research: {Campaign Name}`. The date moves into `metadata.generated_at` and `updated_at`.

### 3. Reuse / skip-or-replace logic before generating

Inside `campaign-agent` (or a small helper called before `performGapResearch`):

- Look up the most recent existing `Campaign Research: {Campaign Name}` doc for this owner.
- If it exists AND `updated_at` is < 30 days old AND `metadata.campaign_focus` + `metadata.target_audience` match the current values → **skip research**, reuse the stored findings as the `researchBlock` context. No new KB row written.
- If focus or audience changed, OR the doc is ≥ 30 days old → run fresh research and **upsert** (update the existing row in place instead of inserting a new one). That guarantees one row per campaign going forward.
- A `force` flag on the request lets the user override and regenerate manually.

### 4. UI surface (minimal)

No new pages. In the KB list, the single remaining "Campaign Research: {Name}" row now shows its focus + audience badges (already supported by the metadata viewer added earlier).

## Technical notes

- Title normalization for the backfill: strip the trailing ` (MM/DD/YYYY)` so old and new title formats group together.
- Deletion uses service-role inside a migration-style data script — RLS not a concern.
- `metadata.source = 'campaign-agent-research'` is already set, so we won't touch Practice Intelligence Reports, brand guidelines, uploads, etc.
- Match key going forward: `(user_id, lower(trim(campaign_name from title)))`.

## Files touched

- `supabase/functions/campaign-agent/index.ts` — stable title, focus/audience in metadata, reuse-or-upsert logic, accept `force`.
- One data cleanup run via the insert tool (DELETE statement) to remove existing duplicates.
- No frontend changes required.
